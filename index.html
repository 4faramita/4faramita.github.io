JavaScript 代码研究论文概览 2013 ~ 2017

2017-07-15

概述

本次收集了近 40 篇近年来关于 JavaScript 分析的论文，从中选出 32 篇进行简单分析。

按会议/期刊区分

  • OOPSLA: 7
  • FSE: 2
  • ICSE: 11
  • IEEE Transactions on Software Engineering：1
  • ASE: 3
  • SOAP(PLDI): 2
  • ESEC/FSE: 5
  • SAS - International Symposium on Static Analysis：1
  • ECOOP - European Conference on Object-Oriented Programming： 1

可见本领域论文主要集中在 ICSE，FSE(ESEC/FSE) 与 OOPSLA。

按时间区分

  • 2017: 5
  • 2016: 7
  • 2015: 12
  • 2014: 3

本次收集的数据集中 2015 年的相关论文最多，2016 与 2017 年虽然（在本次的数据中）数量没有 2015 年多，但是总体上看，这个话题还是十分活跃的

内容源流分解

下面我将基本上以小组 (或某个人带领的团队) 为单位简单解读论文

Saba Alimadadi, Ali Mesbah, Karthik Pattabiraman; University of British Columbia: 动态分析

ICSE’14: Clematis

Understanding JavaScript Event-Based Interactions
作者使用动态分析，对浏览器 JavaScript 的基于事件的行为做了调用依赖分析与时序分析，抽象成 “Scene”，最终以 “电影” 的形式进行可视化。
对于事件驱动的语言，JavaScript 的回调一直令初学者费解。本工具可以使用可视化的手法帮助代码理解。
实现上本文使用动态分析的方法，对代码进行 “插桩”，以获得执行流程等信息。

ICSE’16: Sahand

Understanding Asynchronous Interactions in Full-Stack JavaScript
依然使用上文中的动态分析方法（代码注入），定义客户端与服务端的函数/事件的生命周期以及联系（XHR），之后视觉化。实验部分，统计实验组与对照组对于一份代码在规定时间的理解，实验组只允许使用 Sahand，对照组可以使用任何工具 (Chrome, WebStorm)。规定时间内答题。
重点：动态分析，生命周期定义与视觉化

Anders Møller: 动态分析

ESEC/FSE’13: Server Interface Descriptions for Automated Testing of JavaScript Web Applications (API)

针对客户端 JavaScript 程序对服务端 API 误用的分析。
首先分析服务端开放的 API，然后分析客户端发出的 API，进行比较。
近年这方面的工作不止一篇，但是愚钝如我，认为这个工作的价值并不算太大。即使是作为自动化 Debug 的工具。

OOPSLA’13: Semi-Automatic Rename Refactoring for JavaScript

IDE 的插件开发。使用静态分析的方法对 JavaScript 代码在正确的 scope 内重命名。
难点在于确定正确的 scope确定 “关联” 的同名变量，以达到最小的询问次数。
实验的评判标准是改名的正确性，步骤以及询问用户的次数。

OOPSLA’14: Determinacy in Static Analysis for jQuery

jQuery 静态分析的确定性。
使用了上面 OOPSLA’13 年论文的方法。

OOPSLA’15: Stateless Model Checking of Event-Driven Applications (竞态分析)

关于竞态分析的问题下面会详表。

ICSE’16: Feedback-Directed Instrumentation for Deployed JavaScript Applications (Samsung)

通过记录对象从创建到出错的控制流来除错，重点用于由于某个对象的某个属性定义错误或者被改写而出现的错误，这种错误仅仅使用出错位置和调用栈是不足以方便的发现错误的。
使用 Jalangi 实现
学习点：从 GitHub Issue 收集错误；Path 定义方式

SOAP’17: Systematic Approaches for Increasing Soundness and Precision of Static Analyzers

半自动的 JavaScript 静态分析，基于 delta debugging 和动态分析来提升可靠性和准确度。各个方法都是现成的，本文讲了将它们组合起来使用的经验。
这个叫肥皂的会本次数据中收录了两篇，都相当无聊。

Julian Dolby (IBM): 静态分析

该组坚持不懈的做 JavaScript 静态分析。

ICSE’13: Efficient construction of approximate call graphs for JavaScript IDE services

单纯使用 AST 构建一个 call graph。
依然是为 IDE 的功能添砖加瓦。

ASE’16: Practically Tunable Static Analysis Framework for Large-Scale JavaScript Applications (KAIST)

FSE’16: Revamping JavaScript Static Analysis via Localization and Remediation of Root Causes of Imprecision

使用区域化和「纠正」不精确性的根源来「修补」静态分析
核心问题 => 缩小指向分析的规模和耗时
重点：Point-to Analysis 指向分析。是对代码中每个变量的指向进行分析，得出一个指向图。这对于静态分析来说至关重要。

ICSE’17: Statically Checking Web API Requests in JavaScript (API)

又是一片关于 API 的分析，本文是静态分析，难度稍大，但是依然不表。

KAIST 韩国科学技术院: 静态分析

ASE’15: Static Analysis of JavaScript Web Applications in the Wild via Practical DOM Modeling

ASE’16: Practically Tunable Static Analysis Framework for Large-Scale JavaScript Applications (Julian Dolby)

SOAP’17: Revisiting Recency Abstraction for JavaScript Towards an Intuitive, Compositional, and Efficient Heap Abstraction

ICSE’17: Analysis of JavaScript Web Applications Using SAFE 2.0

易用的 JavaScript app 分析工具 with Pluggability, Extensibility and Debuggability.
工具展示文章，有代码，没有详细阐述。
另一个值得注意的框架：WALA，用于指向分析

Samsung(Samsung Research America, SAS): 动态分析

ESEC/FSE’13: Jalangi

A Selective Record-replay and Dynamic Analysis Framework for JavaScript
一个基于动态分析的 JavaScript 分析工具。
同样通过 “插桩” 的方式，获取每个函数的调用初始状态与离开状态，获得执行 trace。
对于原生方法，获取其执行方法与输入状态；对于非原生，记录其输入输出状态（栈）状态。这样就做到尽量少的记录完整的栈。
最后通过 trace 与依赖图对整个执行过程进行重放。

OOPSLA’15: Selective Control-Flow Abstraction via Jumping

ESEC/FSE’15: MemInsight: Platform-independent Memory Debugging for JavaScript

OOPSLA’15: Static analysis of event-driven Node. js JavaScript applications

ISSTA’15: DLint: Dynamically Checking Bad Coding Practices in JavaScript

SAS’15: SJS: A Type System for JavaScript with Fixed Object Layout

OOPSLA’16: Type Inference for Static Compilation of JavaScript

OOPSLA’16: A Practical Framework for Type Inference Error Explanation

ICSE’16: Feedback-Directed Instrumentation for Deployed JavaScript Applications (Anders Møller)

ICSE’17: Repairing Event Race Errors by Controlling Nondeterminism

控制不确定性来修复事件竞态错误。
事件驱动的一部程序可能会因为事件的执行时序与设计预想不同而产生错误，如方法在定义前被调用，属性在对象创建前被访问，或者多个 AJAX 被执行等。本文通过动态分析代码注入，使用一个事件控制器，根据某些策略来丢弃或推迟（在正确时间重新生成）事件，来保证异步程序按照预想顺序执行。比如以上问题都能用 。application-independent policy 应用程序独立性策略来解决。在屏蔽用户操作的阶段，先是一个等待动画告诉用户暂时不能操作。

通过以上论文可以看到：

  1. 动态分析的方法较为晚上，实用性也更强，但是覆盖率问题可能值得商榷。
  2. 静态分析今年趋向于将多种方法结合，甚至使用动态分析来对静态分析进行补足，但是依然无法做到效率、正确性的完全兼顾。
  3. 动态与静态分析都产生了一批完善的框架，如静态分析的 TAJS, WALA，SAFE；动态分析如 Jalangi。
  4. 韩国的组很强啊

2017-06-22

Description

在图论中，如果一个有向图从任意顶点出发无法经过若干条边回到该点，则这个图是一个有向无环图（Directed Acyclic Graph，DAG）. 对于一个n个节点的有向图（节点编号从0到n-1），请判断其是否为有向无环图.

图的节点数和边数均不多于100000.

请为下面的Solution类实现解决上述问题的isDAG函数，函数参数中n为图的节点数，edges是边集，edges[i]表示第i条边从edges[i].first指向edge[i].second. 如果是有向无环图返回true，否则返回false.

例1：
n = 3，edges = {(0, 1), (0, 2)}，函数应返回true.

Coupons

2017-06-13

Description

美团点评上有很多餐馆优惠券，用户可以在美团点评App上购买。每张优惠券有一个唯一的正整数编号。当用户在相应餐馆就餐时，可以在餐馆使用优惠券进行消费。优惠券的购买和使用按照时间顺序逐行记录在日志文件中，运营人员会定期抽查日志文件看业务是否正确。业务正确的定义为：一个优惠券必须先被购买，然后才能被使用。

某次抽查时，发现有硬盘故障，历史日志中有部分行损坏，这些行的存在是已知的，但是行的内容读不出来。假设损坏的行可以是任意的优惠券的购买或者使用。

现在问这次抽查中业务是否正确。若有错，输出最早出现错误的那一行，即求出最大s，使得记录1到s-1满足要求；若没有错误，输出-1。

输入描述:
m 分别表示 m (1 ≤ m ≤ 5 * 10^5) 条记录。

下面有m行，格式为：

I x （I为Input的缩写，表示购买优惠券x）；

O x（O为Output的缩写，表示使用优惠券x）；

? （表示这条记录不知道）。

这里x为正整数，且x ≤ 10^5 。

输出描述:
-1 或 x(1 ≤ x ≤ m) 其中x为使得1到x-1这些记录合法的最大行号。

输入例子:
0
1
O 1
2
？
O 1
3
I 1
？
O 1
2
I 2
O 1

输出例子:
-1
1
-1
-1
2

Solution

使用字典 log_dic 储存已经遍历的条目。主函数 check() 用来检查当前条目是否合法：
如果是输入，检测字典中是否已有相同未输出的输入，有则报错；
如果是输出，检测字典中是否有相应未输出的输入，没有则报错；
每个 “?” 可以抵消一个错误。

Music Study

2017-06-13

Description

有两段音频，每段音频是一个表示音高的序列。现在想要在第二段音频中找出与第一段音频最相近的部分。

具体地说，就是在第二段音频中找到一个长度和第一段音频相等且是连续的子序列，使得它们的 difference 最小。两段等长音频的 difference 定义为：
difference = SUM(a[i] - b[i])2 (1 ≤ i ≤ n),其中 SUM() 表示求和
其中 n 表示序列长度，a[i], b[i] 分别表示两段音频的音高。现在袋鼠先生想要知道，difference 的最小值是多少？数据保证第一段音频的长度小于等于第二段音频的长度。

输入描述:
第一行一个整数 n (1 ≤ n ≤ 1000)，表示第一段音频的长度。
第二行 n 个整数表示第一段音频的音高（0 ≤ 音高 ≤ 1000）。
第三行一个整数 m (1 ≤ n ≤ m ≤ 1000)，表示第二段音频的长度。
第四行 m 个整数表示第二段音频的音高（0 ≤ 音高 ≤ 1000）。

输出描述:
输出 difference 的最小值

输入例子:
2
1 2
4
3 1 2 4

输出例子:
0

Solution

题目思路很直接，calculate_diff() 计算两个等长数组的 “difference”，针对每一对等长数组，计算这个值即可。

算法概论 8.10

2017-06-10

算法概论 8.10
题目：
8.10. Proving NP-completeness by generalization. For each of the problems below, prove that it is NP- complete by showing that it is a generalization of some NP-complete problem we have seen in this chapter.
(a) SUBGRAPH ISOMORPHISM: Given as input two undirected graphs G and H, determine whether G is a subgraph of H (that is, whether by deleting certain vertices and edges of H we obtain a graph that is, up to renaming of vertices, identical to G), and if so, return the corresponding mapping of V (G) into V (H).
(b) LONGEST PATH: Given a graph G and an integer g, find in G a simple path of length g.
(c) MAX SAT: Given a CNF formula and an integer g, find a truth assignment that satisfies at
least g clauses.
(d) DENSE SUBGRAPH: Given a graph and two integers a and b, find a set of a vertices of G
such that there are at least b edges between them.
(e) SPARSE SUBGRAPH: Given a graph and two integers a and b, find a set of a vertices of G
such that there are at most b edges between them.
(f) SET COVER. (This problem generalizes two known NP-complete problems.)
(g) RELIABLE NETWORK: We are given two n × n matrices, a distance matrix dij and a connec- tivity requirement matrix rij , as well as a budget b; we must find a graph G = ({1, 2, . . . , n}, E) such that (1) the total cost of all edges is b or less and (2) between any two distinct vertices i and j there are rij vertex-disjoint paths. (Hint: Suppose that all dij ’s are 1 or 2, b = n, and all rij ’s are 2. Which well known NP-complete problem is this?)

证明：
(a) 令 G 与 H 顶点数相同，G 为环。若 G 为 H 的同构子图，则 H 中存在 Rudrata 回路。因此该问题是 Rudrata 回路问题的推广。
(b) 图 G 有 V 个顶点，若 g = |V| - 1，则称为求 Rudrata 路径的问题。因此该问题是 Rudrata 路径问题的推广。
(c) 当 g 为字句总数时，该问题变成一个 SAT 问题。因此该问题是 SAT 问题的推广。
(d) 若 b = a * (a - 1) / 2，即 G 中 a 个顶点两两相连，该问题变成一个最大团问题。因此该问题时最大团问题的推广。
(e) 若 b = 0，该问题变成最大独立集问题。因此该问题是最大独立集问题的推广。
(f) 集合覆盖问题：给定全集 U，以及一个包含 n 个集合且这 n 个集合的并集为全集的集合 S。集合覆盖问题要找到 S 的一个最小的子集，使得他们的并集等于全集。
对于任一个顶点覆盖问题 G = 以及整数 q，令全集 U = E，图 G 中的顶点分别标记为 1 到 n，子集 Si 表示所有指向顶点 i 边的集合。这样这个问题就变成了一个集合覆盖问题。因此集合覆盖问题是顶点覆盖问题的推广。
(g) 如提示所言，当所有 d_ij 都为 1 或 2，b = n，所有 r_ij 都等于 2，这个问题就变成了一个 TSP 问题。因此该问题是 TSP 问题的一个推广。

15. 3Sum

2017-06-06

Description

Difficulty: Medium

Given an array S of n integers, are there elements a, b, c in S such that a + b + c = 0? Find all unique triplets in the array which gives the sum of zero.

Note: The solution set must not contain duplicate triplets.

For example, given array S = [-1, 0, 1, 2, -1, -4],

A solution set is:
[
  [-1, 0, 1],
  [-1, -1, 2]
]

题意:

给出一数组，返回数组中和为 0 的三个元素的组合。要求去重。

Solution

一道经典的题。
先考虑 2 sum，针对有序数列，头尾双指针，若当前和大于要求，尾指针前移；若小于要求，头指针后移。时间复杂度 O(n)
那么 3 sum 只要针对每个元素算 2 sum 即可，时间复杂度 O(n ^ 2)。
问题在于去重。首先的想法是用集合，然而集合无法储存数组，因此先将结果数组排序并转换为元组。
然而即使用集合，也是要尝试插入很多次的；另外集合在实现上终究是要查找的，去重不可能没有代价。因此更好得做法是在指针处过滤，若指针当前值与上一个值相等，则跳过当前值。
还有一个点：一开始为了防止越界，我在去重代码部分外面加上了 try，然而无法满足时间要求；去掉 try 语句就可以了。try 的开销比我想象的大。
